# Архитектура проекта

## Назначение документа

Документ описывает техническую архитектуру платформы недвижимости и служит справочником при реализации последующих этапов разработки. Дополняет документы:

- **docs/project-rules.md** — продуктовые правила и ограничения.
- **docs/mvp-scope.md** — границы MVP и объём работ.

---

## Технологический стек проекта

### Backend

- **Python** — язык разработки.
- **Django** — веб-фреймворк и основа backend-приложения.
- **Django REST Framework** — REST API для фронтенда и интеграций.
- **PostgreSQL** — основная СУБД.

### Frontend

- **Next.js** — фреймворк приложения (SSR, маршрутизация).
- **React** — UI-библиотека.
- **TypeScript** — типизация и разработка.
- **TailwindCSS** — планируется для стилизации (базовый стек).

### Инфраструктура

- **Docker** — контейнеризация сервисов.
- **Docker Compose** — оркестрация окружения разработки.

### Карты

- **Yandex Maps** — публичная карта для отображения объектов.

### Медиа

- **Локальное хранилище** — в рамках MVP файлы (изображения объектов и т.п.) хранятся локально на сервере.

Состав стека может уточняться в ходе разработки; перечисленные технологии задают базовые решения для MVP.

## UI и адаптивность интерфейса

Публичный сайт и внутренняя CRM должны иметь **адаптивный интерфейс (responsive UI)**.

Интерфейс должен корректно работать на:

- desktop
- tablet
- mobile

Разработка интерфейса следует принципу **mobile-first**, при котором сначала проектируется мобильная версия, после чего интерфейс масштабируется для планшетов и десктопов.

Для реализации пользовательского интерфейса используется следующий стек:

- React
- Next.js
- TypeScript
- TailwindCSS

Использование **Bootstrap не допускается**, так как проект использует современную компонентную архитектуру на базе React и TailwindCSS.

## Географическая архитектура и структура локаций

Модель локаций проекта изначально ограничена **узким региональным охватом**, а не всей территорией России.

- **Целевой регион первой реализации** — Краснодарский край.
- **Поддерживаемые городские домены** — Краснодар и Геленджик.
- **Иерархия локаций** должна проектироваться под их структуру: районы, микрорайоны, населённые пункты, сёла, деревни, хутора, местности — в соответствии с утверждённой структурой проекта для каждого города.
- **Архитектура не должна предполагать** немедленную поддержку всех городов России или общефедеральный селектор регионов; фильтры и выбор локации ориентированы на утверждённую географию и референсную UX-логику проекта.

Технические модели и структуры данных локаций описываются на этапах реализации; в данном разделе зафиксирован только архитектурный принцип ограниченной географии.

## Структура backend

Ниже описана целевая архитектура backend на уровне модулей. Папки создаются на этапах инициализации Django и модулей, а не в рамках документации.

Планируемая структура:

```
backend/
    core/
    users/
    properties/
    leads/
    media/
    common/
```

### Назначение модулей

| Модуль | Назначение |
|--------|------------|
| **core** | Конфигурация проекта, общие настройки, общая инфраструктура (middleware, базовые классы). |
| **users** | Учётные записи сотрудников, роли, права доступа. |
| **properties** | Объекты недвижимости и связанные с ними данные (типы, параметры, статусы). |
| **leads** | Заявки и формы обратной связи с сайта (запрос по объекту, общая форма и т.п.). |
| **media** | Изображения объектов и обработка медиафайлов. |
| **common** | Общие утилиты, миксины, переиспользуемые компоненты (хелперы, константы, базовые сериализаторы). |

Структура может эволюционировать; описание отражает исходный архитектурный план.

## Структура frontend

Ниже описана целевая архитектура frontend на уровне каталогов. Реализация выполняется на этапах инициализации Next.js и разработки функциональности.

Планируемая структура:

```
frontend/
    app/
    components/
    features/
    lib/
    styles/
```

### Назначение каталогов

| Каталог | Назначение |
|---------|------------|
| **app** | Страницы приложения и маршрутизация (Next.js App Router). |
| **components** | Переиспользуемые UI-компоненты (кнопки, формы, карточки, layout). |
| **features** | Доменные модули фронтенда: каталог, карточка объекта, поиск, фильтры, карта и т.п. |
| **lib** | Общие утилиты, клиенты API, хелперы. |
| **styles** | Глобальные стили и конфигурация Tailwind. |

Структура является плановой и будет реализована на последующих этапах.

## Общий порядок этапов разработки

Разработка ведётся по этапам; каждый этап выполняется последовательно.

| Этап | Содержание |
|------|------------|
| **Этап 1 — Фундамент проекта** | Структура репозитория, инфраструктура разработки (Docker Compose), документация и правила (в т.ч. архитектура, границы MVP). |
| **Этап 2 — Фундамент backend** | Инициализация Django, базовая конфигурация проекта, интеграция с PostgreSQL. |
| **Этап 3 — Базовые модули backend** | Модули users, properties, leads и связанная логика. |
| **Этап 4 — Фундамент frontend** | Инициализация Next.js, базовая UI-обвязка. |
| **Этап 5 — Публичный каталог** | Список объектов, фильтры, поиск. |
| **Этап 6 — Карточка объекта** | Страница объекта, описание, медиа, форма обратной связи. |
| **Этап 7 — CRM** | Функциональность CRM для сотрудников (управление объектами, заявками, пользователями). |
| **Этап 8 — SEO и контент** | SEO-страницы, статьи, контентные разделы. |

Детализация задач по каждому этапу уточняется в планах спринтов и в документации этапов.

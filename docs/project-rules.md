# Правила проекта

## Назначение документа

Документ задаёт единый источник истины по продуктовым решениям и границам текущей версии платформы. Все последующие этапы разработки должны соответствовать этим правилам; выход за рамки допустим только через явно утверждённые новые этапы.

---

## Продуктовые правила

### 1. Сейчас реализуется только продажа недвижимости

В объёме текущего MVP платформа поддерживает исключительно **продажу** объектов недвижимости. Функциональность, связанная с арендой (поиск аренды, типы сделок «аренда», листинги аренды и т.п.), в текущей версии не входит.

### 2. Аренда на текущем этапе не входит в MVP

Аренда явно исключена из текущего объёма работ. Не следует добавлять модели, поля, фильтры или интерфейсы для аренды до тех пор, пока это не будет включено в отдельный этап.

### 3. Публичная регистрация пользователей не реализуется

Самостоятельная регистрация посетителей сайта (форма «Зарегистрироваться», подтверждение email, самосервисное создание аккаунта) не входит в текущий MVP. Не нужно проектировать или реализовывать публичные эндпоинты и страницы регистрации.

### 4. Создание пользователей выполняет только администратор

Учётные записи пользователей (сотрудников, менеджеров и т.д.) создаются только через администрирование — внутренними средствами CRM/админки уполномоченным лицом (администратором). Публичного потока создания пользователей нет.

### 5. Встроенный чат на текущем этапе не реализуется

Функциональность чата между пользователями, клиентами и сотрудниками (в том числе виджеты чата, уведомления в реальном времени, история переписки в рамках платформы) не входит в текущий MVP. Не добавлять чат без утверждённого отдельного этапа.

### 6. Импорт объектов из внешних источников пока не реализуется

Импорт объявлений/объектов из внешних систем (прайс-листы, выгрузки с других площадок, API партнёров и т.п.) не входит в текущий объём. Все объекты в системе добавляются вручную через утверждённые средства платформы.

### 7. Лишние блоки и перегруженные элементы не добавляются

Проект не должен копировать перегруженный функционал крупных маркетплейсов недвижимости (например, CIAN, Avito). Такие площадки допустимы только как **UX-референсы** (подход к фильтрам, карточкам, навигации). Запрещается добавлять блоки, виджеты и тяжёлые сценарии, не утверждённые в текущем объёме MVP (сравнение объектов, избранное в виде отдельной подсистемы, сложные рекомендации, лишние шаги в воронке и т.п.), если они не были явно включены в описание этапа.

---

## Что входит в текущую реализацию

- Публичный сайт недвижимости (каталог, карточки, поиск, фильтры, карта, статьи).
- Внутренняя CRM для сотрудников.
- Ручное управление объектами недвижимости (только продажа).
- Заявки/лиды (формы обратной связи, запросы с сайта).
- Роли и права для персонала.
- Учётные записи пользователей создаются администратором.

---

## Что явно исключено на текущем этапе

- Аренда недвижимости.
- Публичная регистрация пользователей.
- Встроенный чат.
- Импорт из внешних источников.
- Лишние и неутверждённые блоки в стиле крупных маркетплейсов.

---

## Дисциплина разработки

- **Не реализовывать функциональность вне утверждённого объёма.** Если фича не описана в текущем этапе и не входит в docs/mvp-scope.md и project-rules.md — её не добавляют.
- **Каждая новая крупная возможность вводится только через явный этап.** Расширение продукта (аренда, регистрация, чат, импорт и т.д.) должно быть зафиксировано в плане этапов и только после этого попадать в реализацию.
- **Приоритет — понятный и управляемый MVP.** Избегать «на всякий случай» фич и усложнения первой версии без явной необходимости.
